## FGR-BC0201, RLC UM/AM Configuration
### INTRODUCTION
This feature is for RLC parameter configuration. This feature allows an operator to choose the RLC data transfer mode (whether AM mode or UM mode) that is used for each QCI.
### BENEFIT
It provides the operator RLC mode configuration flexibility per QCI.

### FEATURE DESCRIPTION
#### Overview
RLC Sublayer The Radio Link Control (RLC) is a link layer protocol that is located between the PDCP layer and MAC layer and it is applied to the radio link section between UE and gNB. The RLC is working in Acknowledged Mode (AM), Unacknowledged Mode (UM), or Transparent Mode (TM) depending on the service property.
 The AM supports the ARQ function that checks the reception of a packet and the packet re-transmission function to guarantee reliability. It is used for a service where delay-robustness and data reliability are important, such as FTP or an Internet connection.

 The UM has no data re-transmission function, and it is used to quickly send delay-sensitive packets in real-time. The most common application for UM is voice service.
 The TM sends the SDU received from its upper layer without an RLC header to reduce the radio overhead.
The RLC element sends reliable data through data error correction and flow control by interoperating with an upper PDCP element and a lower MAC/PHY. The signaling message sent to the UE from the RRC is transmitted through the Signaling Radio Bearer (SRB) in the TM or AM mode. The user data sent to the DRB through the PDCP is transmitted in the AM or UM mode. The operator can configure which mode will be used for each QCI through the USM. Usually, the default bearer is set to the AM mode, and the bearer that sends a voice packet is set to the UM mode.
Additionally, the RLC provides the packet buffering function. For a UE, if the speed at which the gNB receives from the SGW is higher than the speed at which the gNB transmits through the radio link, packets will be buffered in the RLC of the gNB.
The RLC configuration information, including transmission mode per bearer, is transmitted through the signaling message between the UE and gNB.
The most outstanding features of RLC AM mode for reliable data transmission are the ARQ and re-transmission functions. An AM RLC entity can poll its peer AM RLC entity to trigger STATUS reporting at the peer AM RLC entity, which is done by including the poll flag in the RLC data PDU. The RLC transmitting entity will include the poll flag when there is no data to transmit or when the t-PollRetransmit timer expires or when no new RLC data PDU can be transmitted, for example, due to window stalling. The STATUS reporting includes positive and/or negative acknowledgements of RLC PDUs or portions of them.
The RLC sublayer should follow the NR specification ( 3GPP TS 38.322 and 38.300) and support the following functions.
 Transfer of upper-layer PDUs;
 Error correction through ARQ (only for AM data transfer);
 Segmentation and reassembly of RLC SDUs (only for UM and AM data transfer);
 Re-segmentation of RLC SDU segments (only for AM data transfer);
 Duplicate detection (only for AM data transfer);
 RLC SDU discard (only for UM and AM data transfer);
 RLC re-establishment;
 Protocol error detection (only for AM data transfer)
RLC Mode
An RLC entity can be configured to perform data transfer in one of the following three modes: Transparent Mode (TM), Unacknowledged Mode (UM) or Acknowledged Mode (AM). Consequently, an RLC entity is categorized as a TM

RLC entity, an UM RLC entity or an AM RLC entity depending on the mode of data transfer that the RLC entity is configured to provide.
1) TM RLC entity
A TM RLC entity can be configured to submit/receive RLC PDUs through the following logical channels:
 BCCH, DL/UL CCCH, and PCCH
The following figure depicts a model of two transparent mode peer entities.
[FGR-BC0201_image_1.png]
A TM RLC entity submits/receives the following RLC data PDU:
 TMD PDU
1-1) Transmitting the TM RLC entity
When a transmitting TM RLC entity forms TMD PDUs from RLC SDUs, it should:
 not segment the RLC SDUs;
 not include any RLC headers in the TMD PDUs.
1-2) Receiving the TM RLC entity
When a receiving TM RLC entity receives TMD PDUs, it should:
 Deliver the TMD PDUs (which are just RLC SDUs) to the upper layer.
2) UM RLC entity
A UM RLC entity can be configured to submit/receive RLC PDUs through the following logical channels:
 DL/UL DTCH

The following figure depicts a model of two unacknowledged peer entities.
[FGR-BC0201_image_2.jpg]
An UM RLC entity submits/receives the following RLC data PDU:
 UMD PDU
An UMD PDU contains either one complete RLC SDU or one RLC SDU segment.
2-1) Transmitting UM RLC entity
The transmitting UM RLC entity generates UMD PDUs for each RLC SDU. It should include relevant RLC headers in the UMD PDU. When notified of a transmission opportunity by the lower layer, the transmitting UM RLC entity should segment the RLC SDUs, if needed, so that the corresponding UMD PDUs, with RLC headers updated as needed, fit within the total size of RLC PDUs indicated by lower layer.
2-2) Receiving UM RLC entity
When a receiving UM RLC entity receives UMD PDUs, it should:
 Detect the loss of RLC SDU segments at lower layers;
 Reassemble RLC SDUs from the received UMD PDUs and deliver the RLC SDUs to the upper layer when they are available;
 Discard received UMD PDUs that cannot be reassembled into an RLC SDU due to a loss at lower layers of a UMD PDU that belonged to the particular RLC SDU.

3) AM RLC entity
An AM RLC entity can be configured to submit/receive RLC PDUs through the following logical channels:
 DL/UL DCCH or DL/UL DTCH
The following figure depicts a model of an acknowledged mode entity.
[FGR-BC0201_image_3.png]
An AM RLC entity delivers/receives the following RLC data PDUs:
 AMD PDU
An AMD PDU contains either one complete RLC SDU or one RLC SDU segment.
An AM RLC entity delivers/receives the following RLC control PDU:
 STATUS PDU
3-1) Transmitting side
The transmitting side of an AM RLC entity generates AMD PDUs for each RLC SDU. When notified of a transmission opportunity by the lower layer, the transmitting AM RLC entity should segment the RLC SDUs, if needed, so that the corresponding AMD PDUs, with RLC headers updated as needed, fit within the total size of RLC PDUs indicated by the lower layer.

The transmitting side of an AM RLC entity supports retransmission of RLC SDUs or RLC SDU segments (ARQ):
 If the RLC SDU or RLC SDU segment to be retransmitted (including the RLC header) does not fit within the total size of RLC PDUs indicated by the lower layer at the particular transmission opportunity notified by lower layer, the AM RLC entity can segment the RLC SDU or re-segment the RLC SDU segments into RLC SDU segments.
 The number of re-segmentations is not limited.
When the transmitting side of an AM RLC entity forms AMD PDUs from RLC SDUs or RLC SDU segments, it should:
 Include relevant RLC headers in the AMD PDU.
3-2) Receiving side
When the receiving side of an AM RLC entity receives AMD PDUs, it should:
 Detect whether the AMD PDUs have been received in duplication, and discard duplicated AMD PDUs;
 Detect the loss of AMD PDUs at lower layers and request retransmissions to its peer AM RLC entity;
 Reassemble RLC SDUs from the received AMD PDUs and deliver the RLC SDUs to the upper layer when they are available.
 Adaptive RLC poll retransmission timer for SRB function
This function increases the RLC poll retransmission timer for the SRB adaptively.
This function can be enabled by using the ON/OFF parameter adaptive-rlc-poll-retx-dl-enable ( [DU, UADPF] adaptive-rlc-poll-retransmission ).
If the above parameter is ON, the adaptive timer for SRB can be applied and the timer can be set as follows:
There is a parameter, adaptive-retx-point-dl ( [DU, UADPF] adaptive-rlc-poll-retransmission ), for the time when the adaptive poll timer is applied at retransmission point, and from this point, the poll retransmission timer is applied sequentially according to the timer setting value, adaptive-first-rlc-poll-timer ( [DU, UADPF] adaptive-rlc-poll-retransmission ), adaptive-second-rlc-poll-timer ( [DU, UADPF] adaptive-rlc-poll-retransmission ), and adaptive-last-rlc-poll-timer ( [DU, UADPF] adaptive-rlc-poll-retransmission ). The last timer setting value is maintained for the remaining retransmissions.
For example,
adaptive-retx-point-dl = 28th
adaptive-first-rlc-poll-timer = 90 ms
adaptive-second-rlc-poll-timer = 180 ms

adaptive-last-rlc-poll-timer = 350ms

RLC parameter adaptation for VoNR PUSCH repetition To increase the weak electric field performance of VoNR, a function for repetition PUSCH is provided.
Since this function can increase the total HARQ RTT due to continuous PUSCH transmission, the RLC timers should increase the value considering the effect of HARQ delay.
When VoNR PUSCH repetition feature is supported, the following RLC parameters are configured by different values from YANG configured values if a 5QI=1 bearer is established:
 gnb-timer-poll-retransmit ( [DU, UADPF] drb-rlc-info-func )
 ue-timer-poll-retransmit ( [DU, UADPF] drb-rlc-info-func )
 gnb-t-reassembly ( [DU, UADPF] drb-rlc-info-func )
 ue-t-reassembly ( [DU, UADPF] drb-rlc-info-func )
If 5QI=1 bearer is released, the above value returns to the YANG setting value.
The bearer subject to change is applied to all the DRBs and SRBs that were previously set up at the time of the 5QI=1 setup.
The changed values are calculated as an internal calculation value. The reassembly timer is calculated by considering the time to wait for all HARQ RTTs and the scheduling delay margin. The poll timer is determined by the time to guarantee the reassembly.

Note: DRB RLC parameter can be configured for each QCI or 5QI.

These parameters can be set according to QCI.
For SA, drb-rlc-info-func-5qi ( [DU, UADPF] rlc-functions ) is deployed.
These parameters can be set according to 5QI.

For example, if the operator wants to set DRB RLC parameters corresponding to QCI = 9 and B6G TDD cell, set the RLC parameters in drb-rlc-info-func ( [DU, UADPF] rlc-functions ) where qci ( [DU, UADPF] drb-rlc-info-func ) is 9 and config-type-drb ( [DU, UADPF] drb-rlc-info-func ) is B6G-TDD.

In the RLC parameters, the parameters with the prefix gnb- mean a parameter used in the gNB side, and the parameters with the ue- mean a parameter set in the UE side. Configuration for each gnb/UE is also supported for both DRB and SRB parameters.
For example, gnb-timer-poll-retransmit ( [DU, UADPF] drb-rlc-info-func ) is a timer that waits for poll retransmission when transmitting from the gNB, and ue-timer-poll-retransmit ( [DU, UADPF] drb-rlc-info-func ) is a timer that waits for poll retransmission when transmitting UL from the UE.
