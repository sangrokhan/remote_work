# Week 1 Context

## 왜 이 주차부터 시작하는가
- PRD의 `WorkflowSchema`/UI 계약 선행성이 이번 주의 핵심이다.  
  frontend는 스키마를 기준으로 노드 UI를 구성하므로, 계약이 흔들리면 week2~4 전체 구현비용이 기하급수적으로 증가한다.
- 실행/이벤트 모델을 week02/03에서 빠르게 확장하려면 `workflowId`, `stateKey`, `nodeId`, `edge`의 필드 정합성이 선행 확정되어야 한다.
- 범위/비범위를 미리 분리해 week01에서 계약-조회만 고정하고, 런 상태/이벤트/히스토리 구현은 다음 주로 이관한다.

## PRD와의 정합성
- 범위(3.1)의 `workflow 구조 조회 방식`과 핵심요구(4.1) `workflow schema 조회`를 직접 충족한다.
- 비기능 요구사항(4.1) `backend source-of-truth`, `frontend 하드코딩 최소화`와 정합되며, week04 운영 안정성/오픈이슈 추적이 가능해지는 기반이 된다.
- week02의 run 이벤트/상태 머신도 schema의 `workflow/노드/상태 키`를 참조하도록 강제해 데이터 정합성을 유지한다.

## 구성요소 분해 관점
- 계약 모델: 스키마 스펙, 응답 필드 필수성, 버전/범위.
- 구현 모델: 조회 라우트, 레지스트리, 에러 변환기.
- 운영 모델: 테스트용 fixture, 계약 문서, 변경 이력.

## 아키텍처 정합성 체크 포인트
- 노드 식별자는 UI key와 backend 이벤트 key가 충돌하지 않도록 동일한 네임스페이스 규칙을 따른다.
- schema response의 `metadata`는 phase 간 확장 가능해야 하며, week04 성능/표시 모드 전환에서 파생된 계산 값은 여기서 내려주지 않는다(파생 금지).

## 구현 가능성
- 변경 범위가 정해져 있어 팀 동시 작업 충돌이 낮고, 기존 백엔드 API 구조에 쉽게 통합 가능하다.
- 주차 내 산출물은 코드+테스트+문서로 완료 가능하고, 다음 주차 연결점은 명확한 인터페이스 계약으로 관리.
